# Zero-trust для воркеров (ЭТАП 23.5)

Воркеры (DuoPlus/Playwright/FFmpeg) — самые опасные компоненты: исполняют браузеры и
команды на удалённых устройствах. Принцип: ничему не доверяем по умолчанию, минимум прав,
ничего лишнего наружу.

## Сеть
- Приватная сеть web↔worker↔Redis↔DB через **Fly private networking** (6PN) или Tailscale.
- Воркеры **без публичных портов**. Наружу — только `/health` (4001) за внутренним LB,
  не доступный из интернета.
- БД/Redis принимают коннекты только из приватной сети (Supabase: allowlist; Upstash: TLS+токен).

## Аутентификация сервис-сервис
- Внутренние вызовы (web→worker enqueue идёт через Redis, не HTTP) — по умолчанию через очередь.
- Где нужен прямой внутренний HTTP — **подписанные запросы** (HMAC общим секретом из Doppler)
  или **mTLS** (клиентские сертификаты, выдаются на деплое).
- Никаких внутренних эндпоинтов без аутентификации.

## Least-privilege
- **Секреты**: воркер получает scoped-конфиг Doppler (только нужные ключи: REDIS_URL, DB,
  AI/PHONE/PROXY — НЕ платёжные/CAPTCHA секреты web).
- **БД**: воркер ходит под ролью с минимальными правами; `service_role` (BYPASSRLS) —
  только там, где реально нужно (системные джобы), не по умолчанию.
- **Устройства/прокси**: креды расшифровываются в момент использования (ЭТАП 2/3),
  не задерживаются в памяти/логах.

## Изоляция исполнения
- FFmpeg/Playwright — в контейнере воркера (ЭТАП 18.2), non-root, dumb-init.
- Боевая версия sandbox (ЭТАП 6.8): ограничение ресурсов (cgroups), `--no-network` для
  процессов рендера, где сеть не нужна; раздельные пулы cpu/browser.
- Входные медиа валидируются (ffprobe-лимиты ЭТАП 6.2) до обработки.

## Проверка
- Воркер недоступен из интернета (скан портов снаружи → только закрыто/health за LB).
- Внутренняя связь — по приватной сети + подпись/mTLS.
- Аудит доступа к секретам (Doppler access log).
