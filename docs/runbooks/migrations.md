# Runbook: миграции БД (Drizzle)

## Принципы

- **Только forward.** Откатных down-миграций в проде нет. Откат = новая forward-миграция.
- **Никогда не редактировать сгенерированные SQL вручную.** Меняешь схему (`packages/db/src/schema/`) → генерируешь новую миграцию.
- **Разрушающие изменения — через expand-contract:**
  1. _expand_ — добавить новое (колонку/таблицу), не ломая старое;
  2. _backfill_ — перенести данные;
  3. _switch_ — переключить чтение/запись на новое;
  4. _contract_ — отдельной поздней миграцией удалить старое.

## Рабочий цикл

```bash
# 1. изменить схему в packages/db/src/schema/index.ts
pnpm db:generate     # сгенерирует migrations/NNNN_*.sql (БД не нужна)
# 2. просмотреть сгенерированный SQL глазами (он коммитится в репозиторий)
pnpm db:migrate      # применить к локальной БД (нужен запущенный Postgres: pnpm db:up)
pnpm db:studio       # визуально проверить
pnpm db:seed         # наполнить тестовыми данными (идемпотентно)
```

## Команды

- `pnpm db:generate` — SQL из схемы (без БД).
- `pnpm db:migrate` — применить миграции (нужна БД).
- `pnpm db:studio` — Drizzle Studio (нужна БД).
- `pnpm db:seed` — seed-данные (нужна БД; грузит `.env` через `--env-file`).

## Seed

`packages/db/src/seed.ts` идемпотентен: сначала удаляет прошлый seed (org с фиксированным id +
seed-пользователи по email), затем вставляет: 1 org, 2 пользователя (owner+editor), 3 соц-аккаунта,
2 медиа, 1 подписку. Повторный запуск даёт тот же результат.

## Прод

- Миграции применяются отдельным шагом CI/деплоя (ЭТАП 16), не автоматически при старте приложения.
- Бэкап перед миграцией (PITR + dump — ЭТАП 16).
