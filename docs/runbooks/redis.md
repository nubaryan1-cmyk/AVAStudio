# Runbook — Redis prod (Upstash) + инциденты (ЭТАП 18.4)

Redis держит очереди BullMQ (рендер, постинг, варминг, AI) и DLQ. Эксперт отметил его
как single point of failure — отсюда HA, алерты и проверка деградации.

## Подключение
- **Upstash Redis** (prod), регион рядом с воркерами (`fra`), **persistence включён**.
- `REDIS_URL` (с TLS, `rediss://`) → Doppler prod. Приложение и воркеры читают из env.
- Пул соединений: переиспользовать одно подключение на процесс (уже в `@avastudio/queue`).

## HA
- Включить Upstash replication/global для отказоустойчивости чтения.
- Алерты (Upstash console / BetterStack, ЭТАП 24): доступность, latency p99, used-memory,
  eviction count, число коннектов.

## Поведение при инциденте Redis
Очереди спроектированы на устойчивость (ЭТАП 5.4 DLQ + replay, ЭТАП 8 resilience):
- Воркер не падает при недоступности Redis — health на 4001 отдаёт `degraded/down` (503),
  LB перестаёт слать трафик, машина не убивает задачи.
- Продюсеры (web) при ошибке enqueue возвращают понятную ошибку, не теряют пользовательский ввод.
- После восстановления Redis — задачи продолжают обрабатываться; зависшие → DLQ → replay.

## Чеклист при инциденте
1. Проверить статус Upstash (console) и алерты.
2. Проверить `/health` воркеров (4001) — `down` указывает на Redis.
3. Если Upstash деградировал — переключить `REDIS_URL` на резервный инстанс в Doppler, передеплоить.
4. После восстановления — проверить DLQ (`listDLQ`) и сделать replay застрявших задач.
5. Зафиксировать инцидент и причину.

## Проверка устойчивости (drill)
- На staging: временно сделать `REDIS_URL` недоступным → убедиться, что задачи не теряются,
  воркеры переживают, после возврата Redis backlog разгребается.
