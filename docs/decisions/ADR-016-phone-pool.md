# ADR-016: PhonePool — провайдеро-независимый пул облачных телефонов

**Статус:** принято (ЭТАП 12, TASK 12.2, Фаза 1 — mock)

## Контекст
Часть соцсетей (Instagram, TikTok) требует постинга с мобильных устройств. Сервисы
облачных телефонов (DuoPlus и аналоги — GeeLark, MoreLogin, BlueStacks Cloud) дают
арендуемые Android-устройства с ADB/SDK. Привязка к одному провайдеру — риск: блокировка
аккаунта у вендора, рост цен, простой при сбое вендора, гео-ограничения.

## Решение
КЛЮЧЕВОЕ: НЕ привязываемся к DuoPlus. Вводим абстракцию `PhoneProvider` (интерфейс) и
оркестратор `PhonePool`. Минимум два провайдера реализуют интерфейс сразу (DuoPlus +
аналог), что гарантирует отсутствие «протекания» специфики одного вендора в код.

`PhoneProvider`: `isHealthy`, `rentDevice`, `releaseDevice`, `listDevices`, `installApp`,
`executeAction`, `getStatus` + `capabilities` (adb/screenshot/appInstall/maxDevices).
Устройство нормализуется в `Device` (id/provider/state/region/androidVersion).

`PhonePool` поверх упорядоченного списка провайдеров:
- **failover A→B** — аренда идёт по первому здоровому провайдеру; недоступный/упавший
  пропускается, ошибки агрегируются, при исчерпании всех — единая ошибка.
- **пулинг/re-use** — освобождённые устройства кэшируются по `label` (аккаунт) и
  переиспользуются, экономя время аренды и снижая «новизну» устройства (анти-бан).

## Фазы
- Фаза 1 (сейчас): интерфейс + in-memory mock-драйверы (`duoplus`, `geelark`), без
  реальных подключений и ключей.
- Фаза 2 (ЭТАП 22): реальные SDK-драйверы поверх того же интерфейса, без изменения
  PhonePool и вызывающего кода.

## Последствия
+ Смена/добавление провайдера = новый драйвер, остальной код не трогаем.
+ Устойчивость к сбою вендора (failover) и оптимизация затрат (re-use).
− Нормализованная модель `Device` беднее нативных SDK; редкие фичи — через capabilities.

## Альтернативы
Прямой вызов DuoPlus SDK — отклонено (vendor lock-in, единая точка отказа).
