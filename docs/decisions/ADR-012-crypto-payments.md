# ADR-012 — Крипто-платежи: выбор провайдера

Дата: 2026-05-30
Статус: принято (Фаза 1 — stub)
Этап: ЭТАП 9, TASK 9.4

## Контекст
Продукт требует оплату криптовалютой наравне с картами. Крипто-драйвер обязан реализовать тот же интерфейс `PaymentProvider` (ADR-011), чтобы ядро не различало способ оплаты.

## Варианты
- **NOWPayments** — широкий выбор монет (300+), готовый hosted-invoice, IPN-вебхуки с HMAC-подписью, конвертация фиат→крипта по курсу провайдера. Минус — кастодиальный, KYC у провайдера.
- **Coinbase Commerce** — простая интеграция, но ограниченный набор монет и регионов; будущее развитие продукта под вопросом.
- **BTCPay Server** — self-hosted, без комиссий и кастодиальности, максимальная приватность. Минус — нужно держать инфраструктуру (нода/хостинг), сложнее в Фазе 1.

## Решение
В Фазе 1 ориентируемся на **профиль NOWPayments** (широкий выбор монет + простой IPN). Драйвер написан абстрактно (stub), поэтому фактический провайдер подтверждается в Фазе 2 (ЭТАП 19) — при желании заменяется на BTCPay тем же интерфейсом без правки ядра. Решающий критерий выбора NOWPayments — покрытие монет и наименьшее трение интеграции для гибридного B2C+B2B.

## Как смоделировано
- `createCheckout` → invoice: адрес кошелька + сумма в выбранной крипте (конвертация фиат→крипта через подключаемый `RateProvider`; в Фазе 1 — детерминированные стаб-курсы). Поля `cryptoAddress`/`cryptoNetwork`/`expiresAt` в `CheckoutSession`.
- `parseWebhook` → нормализованное событие. `payment_succeeded` по статусу `finished` либо `confirmed/sending` при достижении `requiredConfirmations` (N подтверждений сети). `failed/expired/refunded` → `payment_failed`. Промежуточные статусы (`waiting/confirming/partially_paid`) — не терминальные, отклоняются. Подпись IPN — HMAC-SHA512 над payload (схема NOWPayments).
- **Нет recurring.** `capabilities.recurring=false`. Подписка моделируется как периодические invoice; драйвер отдаёт `nextRenewalDate()` и `isRenewalDue()` для renewal-напоминаний (плановая постановка нового invoice — на стороне приложения/cron).
- `Money` поддерживает крипто (`kind: "crypto"`, тикер в `currency`, сумма строкой для точности).

## Последствия
- Один и тот же `PaymentProvider`-интерфейс для карт и крипто → единый webhook-handler (TASK 9.5).
- Идемпотентность по `payment_id` (event.id) — дедуп в обработчике.
- Реальные ключи и hosted-invoice — Фаза 2; смена провайдера (BTCPay/Coinbase) не затрагивает ядро.
