# ADR-017: ProxyManager — провайдеро-независимый менеджер sticky-прокси

**Статус:** принято (ЭТАП 12, TASK 12.3, Фаза 1 — mock)

## Контекст
Каждому соц-аккаунту нужен свой стабильный residential proxy (sticky session): смена
IP у «горячего» аккаунта или общий IP на много аккаунтов → checkpoint/бан. Провайдеры
(Bright Data, IPRoyal, Smartproxy) различаются API; привязка к одному — vendor lock-in
и единая точка отказа. Креды прокси — секрет, их нельзя хранить/логировать открыто.

## Решение
Абстракция `ProxyProvider` (интерфейс) + оркестратор `ProxyManager`.

`ProxyProvider`: `isHealthy`, `acquireSticky(spec)→ProxySession`, `release(sessionId)`
+ `capabilities` (sticky/residential/protocols). Сессия несёт стабильный endpoint и
креды (используются сразу, не задерживаются в памяти).

`ProxyManager` поверх списка провайдеров:
- **sticky per account** — `assign` для уже привязанного аккаунта возвращает ту же
  запись (НЕ меняем IP); новый IP только через явный `rotate` (бан/деградация).
- **failover** — недоступный провайдер пропускается, переход к следующему.
- **шифрование кредов** — перед сохранением `encryptJSON(creds, DEK)` (ADR-009,
  ЭТАП 2); в `StoredProxy` лежит `EncryptedBlob`, не открытый пароль.
- **reputation** — счётчики success/fail per proxy (заготовка; scoring — ЭТАП 25).

Хранение — порт `ProxyRepository` (in-memory в Фазе 1, БД-адаптер — ЭТАП 4).

## Фазы
- Фаза 1: интерфейс + in-memory mock-драйвер + InMemoryProxyRepository.
- Фаза 2 (ЭТАП 25): реальные драйверы провайдеров + полноценный reputation-scoring,
  без изменения ProxyManager и вызывающего кода.

## Последствия
+ Смена/добавление провайдера = новый драйвер; sticky-логика и шифрование не меняются.
+ Устойчивость к сбою вендора (failover), безопасное хранение кредов.
− Нормализованная модель беднее нативных опций провайдеров; редкое — через capabilities.

## Альтернативы
Прямой вызов одного API провайдера — отклонено (lock-in, единая точка отказа,
риск утечки кредов без слоя шифрования).
