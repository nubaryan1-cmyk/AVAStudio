# ADR-011 — Провайдеро-независимая абстракция платежей

Дата: 2026-05-30
Статус: принято
Контекст этапа: ЭТАП 9 (Платежи, Фаза 1 — локально, без реальных ключей)

## Контекст
AVAStudio — гибрид B2C+B2B. По требованию продукта оплата возможна и картами, и криптой, и любыми будущими провайдерами. Привязка ядра к Stripe недопустима: смена/добавление провайдера не должны затрагивать бизнес-логику (подписки, лимиты, usage).

## Решение
Вводим единый интерфейс `PaymentProvider` (`packages/shared/src/payments/types.ts`). Каждый способ оплаты — отдельный **драйвер**, реализующий этот интерфейс. Ядро работает только с нормализованными типами и не знает о конкретном провайдере.

### Какие операции абстрагируем
- `createCheckout(plan, org, opts)` → `CheckoutSession` (для карт — redirect URL, для крипто — адрес/invoice).
- `getSubscription(id)`, `cancelSubscription(id)`, `changePlan(id, newPlan)` → `ProviderSubscription`.
- `parseWebhook(payload, signature)` → `NormalizedPaymentEvent` (проверка подписи + маппинг).
- `createPortalSession(customerId)` — опционально (только если `capabilities.portal`).
- `capabilities` — флаги `recurring` / `portal` / `refunds`, чтобы ядро не вызывало неподдерживаемое.

### Нормализованная модель (карты ≡ крипто на уровне ядра)
- `NormalizedPaymentEvent` с унифицированными типами: `subscription_created` / `subscription_updated` / `subscription_cancelled` / `payment_succeeded` / `payment_failed`. Поле `id` уникально у провайдера → ключ идемпотентности (используется в TASK 9.5).
- `Money` — `amount` десятичной **строкой** (а не number) + `currency` (ISO-4217 для фиата или тикер для крипты) + `kind: fiat|crypto`. Строка выбрана, чтобы не терять точность на крипте (8+ знаков) и копейках.
- `CheckoutSession` содержит общие поля + опциональные крипто-поля (`cryptoAddress`, `cryptoNetwork`), чтобы один тип покрывал оба способа.

### Как маппятся разные провайдеры
Каждый драйвер в `parseWebhook` переводит своё событие (Stripe `checkout.session.completed`, крипто `payment.confirmed` и т.п.) в `NormalizedPaymentEvent`. `providerPriceId` (id цены у провайдера) приходит из маппинга план↔price (TASK 9.2), не хардкодится в ядре. `metadata` (например `orgId`) протаскивается в checkout и возвращается в webhook — так событие привязывается к организации без хранения соответствий у провайдера.

### Выбор активного провайдера
`PaymentProviderRegistry` хранит драйверы по имени (домен `PAYMENT_PROVIDERS`). `selectActiveProvider(registry, preferred?)` выбирает активный по имени из конфигурации приложения (env читается на стороне приложения, не в ядре) или первый зарегистрированный.

## Изоляция модуля
Подсистема экспонируется подпутём `@avastudio/shared/payments`, а НЕ через корневой barrel. Причина: в домене уже есть тип `PaymentProvider` — union имён провайдеров (`"stripe" | "crypto" | ...`). Интерфейс `PaymentProvider` (контракт драйвера) — другое понятие. Держим их в разных пространствах имён (как уже сделано для `./telemetry`, `./resilience`), чтобы избежать коллизии символов и не тянуть платёжный слой в браузерный бандл.

## Последствия
- Добавить способ оплаты = новый драйвер, реализующий интерфейс; ядро не меняется.
- Тестируемость: `MockPaymentDriver` реализует тот же контракт → бизнес-логику можно тестировать без сети и ключей.
- Фаза 1 — только код + stub/test (драйверы 9.3/9.4); реальные ключи и live-режим — Фаза 2 (ЭТАП 19).

## Альтернативы
- Прямая интеграция со Stripe SDK в ядре — отклонено: vendor lock-in, крипто не ложится в модель Stripe.
- Готовый биллинг-фреймворк — отклонено в Фазе 1: избыточно, не покрывает крипто-инвойсы единообразно.
