# ADR-009: Архитектура секретов (два контура + envelope encryption)

## Status

Accepted — 2026-05-22

## Context

AVAStudio оперирует двумя принципиально разными типами чувствительных данных:

1. **App-secrets** — секреты самого приложения: строки подключения к БД и Redis, ключи
   платёжных/AI/телефонных провайдеров, ключ шифрования. Их немного, они общие для всего
   инстанса, меняются редко, и к ним обращается только серверный код.

2. **User-credentials** — секреты пользователей: пароли соц-аккаунтов, access/refresh-токены,
   cookies сессий, креды прокси. Их МНОГО (растёт линейно с числом аккаунтов, цель — миллионы),
   они принадлежат конкретной организации, и компрометация одной записи не должна раскрывать
   остальные.

Смешивать эти контуры опасно: у них разные модели угроз, разные места хранения, разный жизненный
цикл и разные требования к ротации. Нужно зафиксировать архитектуру до написания кода (TASK 2.2-2.4),
потому что она определяет схему БД и её потом дорого менять.

Ключевой вопрос для user-credentials: **как организовать шифрование** — один мастер-ключ на все
данные или многоуровневая схема (envelope encryption)?

## Decision

### Контур 1 — App-secrets

- Источник истины: переменные окружения. Локально — `.env` (в `.gitignore`), в проде — Doppler
  (ЭТАП 15), инжектящий значения в `process.env`.
- Доступ — ТОЛЬКО через типизированный модуль `@avastudio/shared/env` (TASK 2.2): Zod-валидация
  при старте, падение процесса при отсутствии/невалидности. Прямой `process.env` в коде запрещён
  ESLint-правилом.
- В репозиторий секреты не коммитятся никогда; есть только `.env.example` с пустыми значениями.

### Контур 2 — User-credentials: **envelope encryption** (выбрано)

Двухуровневая схема ключей:

- **KEK (Key Encryption Key)** — мастер-ключ. App-secret `CREDENTIALS_ENCRYPTION_KEY`, живёт в
  Doppler/`.env`, НИКОГДА не попадает в БД. В будущем выносится в управляемый KMS (AWS/GCP KMS)
  без изменения схемы данных.
- **DEK (Data Encryption Key)** — ключ данных, свой на каждую организацию. DEK генерируется
  случайно, шифруется KEK'ом и хранится «обёрнутым» (wrapped) в БД (таблица ключей).
- Сами креды (пароли/токены/cookies) шифруются соответствующим DEK организации алгоритмом
  **AES-256-GCM** (уникальный IV на каждое шифрование, аутентификация через GCM-tag).
- Версионирование ключей (`keyVersion`) заложено для ротации (TASK 2.4).

### Принцип безопасности (инвариант)

- Компрометация **только БД** (без KEK) → на руках лишь зашифрованные DEK и шифртексты — бесполезны.
- Компрометация **только KEK** (без БД) → расшифровывать нечего.
- Раскрытие **одного DEK** → пострадает максимум одна организация, не вся платформа.

### Политика доступа

| Компонент                             | App-secrets           | User-credentials (расшифровка)                              |
| ------------------------------------- | --------------------- | ----------------------------------------------------------- |
| Frontend (Server Components / Client) | только NEXT*PUBLIC*\* | НИКОГДА                                                     |
| API-роуты (tRPC)                      | да (нужные)           | нет (только запись через репозиторий, без чтения plaintext) |
| Worker'ы (FFmpeg/постинг/телефоны)    | да (нужные, scoped)   | ДА — только здесь, для автоматизации                        |
| Логи / трассировка                    | redaction (TASK 2.4)  | redaction — никогда в plaintext                             |

Расшифровка user-credentials доступна исключительно серверной автоматизации (worker'ы под
service-role), но не возвращается в обычных API/UI-запросах.

## Alternatives Considered

1. **Single master-key (один ключ шифрует все записи напрямую).**
   Проще в реализации. Отклонено: ротация мастер-ключа = перешифровка ВСЕХ записей (миллионы строк);
   нет изоляции (утечка ключа = вся платформа); GDPR-удаление org требует физического удаления строк,
   а не «крипто-шреддинга».

2. **Per-record key (свой ключ на каждую запись).**
   Максимальная изоляция, но избыточно: гигантское число ключей, сложное управление, оверхед.
   Гранулярности на уровне организации достаточно.

3. **Только внешний KMS (шифровать каждый вызов через AWS/GCP KMS).**
   Сильная безопасность, но привязка к облаку с первого дня (против нашего принципа provider-agnostic),
   стоимость и latency на каждый вызов, не работает в локальной Фазе 1. Envelope позволяет начать с
   локального KEK и позже перенести KEK в KMS без переписывания.

**Почему envelope:** единственный вариант, делающий ротацию ключей и GDPR-крипто-шреддинг
выполнимыми на масштабе 1M+, с изоляцией по организациям и путём миграции на KMS.

## Consequences

### Positive

- Ротация KEK перешифровывает только небольшое число DEK, а не все данные.
- Утечка DEK ограничена одной организацией.
- Крипто-шреддинг: удаление DEK org → данные org мгновенно нечитаемы (быстрый GDPR-delete).
- KEK можно вынести в управляемый KMS позже, не трогая данные.
- Чёткая изоляция от app-secrets и от frontend.

### Negative

- Сложнее single-key: нужна таблица обёрнутых DEK и логика wrap/unwrap.
- Кэширование расшифрованных DEK в памяти worker'а требует аккуратности (TTL, очистка).

### Risks

- Потеря KEK = потеря всех данных (нет recovery). Митигация: безопасное хранение KEK + бэкап KEK
  отдельно от БД, документированная процедура (TASK 2.4 + ЭТАП 15).
- Ошибка в реализации wrap/unwrap. Митигация: тесты round-trip и negative-кейсы (TASK 2.3).
