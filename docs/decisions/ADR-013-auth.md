# ADR-013 — Аутентификация: абстракция AuthProvider + локальный драйвер

Дата: 2026-05-30
Статус: принято
Этап: 10 (TASK 10.1)

## Контекст
Прод-аутентификация планируется на Supabase Auth (Фаза 2). В Фазе 1 (локально) нужен
рабочий auth без привязки к облаку и без поднятия Supabase в Docker на каждой машине.

## Решение
Вводим интерфейс `AuthProvider` (signUp/signIn/signOut/getSession/verifyEmail) и
**локальный драйвер** `LocalAuthProvider` сейчас. Supabase-драйвер — в Фазе 2, реализует
тот же интерфейс, вызывающий код не меняется.

Рассмотренные альтернативы:
- **Supabase CLI local (Docker).** Отклонено для Фазы 1: тяжёлая локальная зависимость,
  завязка тестов на внешний сервис. Остаётся вариантом для интеграционных прогонов.
- **Хардкод Supabase SDK.** Отклонено: vendor lock-in, противоречит провайдеро-независимой
  архитектуре проекта (ср. ADR-011 для платежей).

## Детали локального драйвера
- Пароли: **bcrypt** (`bcryptjs`, pure-JS, cost=12). Открытый пароль не хранится/не логируется.
- Сессии: **HS256 JWT** через `node:crypto` (без новых зависимостей). access-токен с `exp`.
  Полноценные сессии (refresh/revoke/cookie) — TASK 10.3.
- Хранение пользователей абстрагировано портом `UserStore` (in-memory фейк в тестах,
  реальная БД — в инфраструктуре).
- **Связка с RLS:** `rlsContextSql(userId)` → `SET app.current_user_id = '...'` (с валидацией
  против инъекций); `applyRlsContext` выполняет на каждый запрос от имени пользователя.

## Последствия
- Единый интерфейс → переход на Supabase в Фазе 2 без переписывания вызовов.
- bcrypt — синхронно-тяжёлый (cost=12): приемлемо для Фазы 1, в Фазе 2 можно вынести/настроить.
