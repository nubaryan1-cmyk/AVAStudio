# Бизнес-метрики и план найма (TASK 26.5)

Финансовый контроль + триггеры масштабирования. Расчёты — `@avastudio/shared/growth/finance`.

## Ключевые метрики
| Метрика | Определение | Источник |
|---------|-------------|----------|
| MRR     | сумма активных подписок/мес | платежи (карты+крипто, ЭТАП 19) |
| ARR     | MRR × 12 | `arr()` |
| Churn   | отток / база на начало периода | `churnRate()` |
| ARPU    | MRR / число подписчиков | `arpu()` |
| LTV     | ARPU × маржа / месячный churn | `ltv()` |
| CAC     | маркетинг-расходы / привлечённые | `cac()` |
| LTV/CAC | здоровье юнит-экономики | `ltvCacRatio()` |
| Рост MoM| (тек−пред)/пред | `momGrowth()` |
| Маржа   | выручка − стоимость обслуживания | unit-economics (25.6) |

## Единый финансовый отчёт
Свести данные обоих платёжных провайдеров (карты+крипто) в один отчёт. Источник —
нормализованные платёжные события (ЭТАП 9.5). Маржа — из unit-economics (cost-per-user, 25.6).

## Триггеры (правила)
- **LTV/CAC > 3** → масштабировать маркетинг (здоровая экономика).
- **Churn растёт** → приоритет retention/онбординг, не привлечение.
- **Убыточные сегменты** (`unprofitableUsers`, 25.6) → пересмотр тарифа/лимитов.

## План найма (с триггерами)
1. **Senior Full-stack** — сразу после стабильного запуска (продуктовая скорость).
2. **DevOps/SRE** — при >50 воркеров или росте инцидентов.
3. **Support** — при >300 платящих (нагрузка на поддержку).
4. **Designer** — когда продуктовый поток требует системного UX.

Каждой роли — runbook онбординга (доступы least-privilege, ключевые runbooks, кодовая база).

## Дашборд
Финдашборд (на ClickHouse, 25.4): MRR/ARR/churn/LTV/CAC/маржа/рост MoM в реальном времени,
топ-затратные юзеры, маржинальность тарифов. Алерт на убыточные сегменты.
